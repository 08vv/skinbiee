A) SYSTEM ARCHITECTURE (CURRENT)
================================

1) Product OCR + Analysis path (active)
--------------------------------------
Frontend:
- `frontend/skinbiee.js`
- Sends product image + user_id to backend endpoint:
  `POST /api/analyze-product`

Backend API:
- `analysis_server.py`
- Endpoint: `/api/analyze-product`
- Steps:
  1. validate `user_id`
  2. read image bytes from multipart upload
  3. upload image to Cloudinary
  4. run EasyOCR on image bytes
  5. filter OCR text by confidence
  6. pass extracted string to ingredient scoring engine
  7. save scan metadata in DB
  8. return JSON to frontend

Scoring engine:
- `modules/product_scanner.py`
- `modules/recommendations.py`
- data source: `data/ingredient_db.json`
- This is rule-based matching (not learned NLP).


2) NLP chatbot path (active, rule-based)
----------------------------------------
Two places exist:

A. Main web app chatbot:
- `frontend/skinbiee.js`
- Function `getMascotAIResponse(input)`
- Rule logic: keyword checks + template responses in JS
- No network LLM call

B. Streamlit mascot chatbot module:
- `modules/mascot_chatbot.py`
- Loads intents from `data/chat_data.json`
- Matches keywords against user text
- Returns random response from matched intent
- Also no network LLM call

Summary:
- NLP exists, but it is deterministic keyword/intent routing.
- No embeddings/vector DB/RAG/transformer inference in current runtime path.


3) OpenRouter path (configured but NOT wired)

Conclusion:

B) OCR PIPELINE DETAILS (LOW LEVEL)
===================================

Primary runtime OCR engine
--------------------------
In `analysis_server.py`:
- Global reader initialization:
  `reader = easyocr.Reader(['en'], gpu=False)`

Implications:
- English model only
- CPU-only inference
- Reader initialized once at process start (good for latency vs per-request init)

OCR execution in active endpoint
--------------------------------
In `/api/analyze-product`:
- `ocr = reader.readtext(b)` where `b` is raw uploaded image bytes
- Extracted text concatenation:
  `txt = ", ".join([t for (bb,t,p) in ocr if p > 0.2])`

Returned OCR payload:
- `ingredients` field = concatenated OCR text string
- raw bounding boxes/confidence are not returned to client


Helper OCR module
-----------------
`modules/ocr_utils.py` has reusable OCR function:
- `extract_ingredients_from_image(image_bytes)`
- It creates its own reader inside function
- It converts bytes -> PIL -> numpy array
- Uses confidence threshold `prob > 0.3`

Important implementation note:
- This helper is currently NOT called by `/api/analyze-product`.
- Active endpoint performs OCR inline in `analysis_server.py`.


C) NLP DETAILS (WHAT YOU THOUGHT WAS MISSING)
=============================================

Yes, NLP part exists.
---------------------
But it is rule-based intent matching, not LLM-driven NLP.

1) Intent source
- `data/chat_data.json`
- Each intent has:
  - `keywords[]`
  - `responses[]`

2) Matching style
- Lowercase input
- keyword contains check
- first/any matched intent decides response (implementation-dependent)
- fallback response if no intent matches

3) Personalization tokens
- Responses include placeholders like `{name}`, `{condition}`, `{streak}`
- Filled from runtime state/session memory

4) Operational behavior
- deterministic + fast + no external API cost
- limited semantic understanding compared to LLMs


D) OCR + NLP + RULE ENGINE INTERACTION
======================================

End-to-end for product scan:
1. OCR extracts text from product label image
2. Rule engine scores ingredient compatibility
3. Frontend presents recommendation + details
4. User may ask chatbot questions
5. Chatbot answers via keyword intents, not via model-generated reasoning

So this subsystem is hybrid:
- OCR = ML model inference (EasyOCR)
- Ingredient "analysis" = rules/heuristics
- Chat "NLP" = intent-keyword templates


E) DISCREPANCY FOUND (DOCS vs ACTUAL CODE)

This is why documentation can feel inconsistent.

G) CURRENT LIMITATIONS
======================

OCR limitations
---------------
- No explicit preprocessing (deskew/denoise/contrast) before OCR
- Single confidence threshold (0.2) hardcoded
- No raw OCR debug payload to inspect false positives
- English-only OCR model

NLP limitations
---------------
- Keyword matching can miss paraphrases
- No context window/memory beyond simple state
- No factual grounding or uncertainty handling typical of LLM flows

Architecture limitations
------------------------
- Duplicate OCR implementation paths (inline vs helper module)
- Product analysis condition currently hardcoded to `"acne"` in endpoint

1. CURRENT PRODUCT SCAN FLOW
============================

Active frontend path
--------------------
- `frontend/skinbiee.js` sends a multipart request to:
  `POST /api/analyze-product`
- The frontend expects:
  - `status`
  - `skin_condition`
  - `ingredients_detected`
  - `ingredient_breakdown`
  - `analysis`
  - `image_url`

Active backend path
-------------------
In `analysis_server.py`, `/api/analyze-product` now does this:
1. Validate `user_id`
2. Read uploaded image bytes
3. Upload the image to Cloudinary (`skinbiee/product_scans`)
4. Run OCR through `modules/ocr_utils.extract_ingredients_from_image(...)`
5. Parse the ingredient section with `modules/ocr_utils.extract_ingredient_section(...)`
6. Fetch the user's latest detected skin condition from the database via
   `get_user_skin_condition(user_id)`
7. Try LLM ingredient analysis through `modules/llm_provider.py`
8. Fall back to rule-based scoring with `modules/product_scanner.py` if LLM fails
9. Build a per-ingredient breakdown
10. Save the scan in the `scans` table as `condition = "Product Scan"`
11. Return structured JSON to the frontend

This means the older description that said the endpoint did inline OCR and always
used hardcoded `"acne"` is no longer correct.


2. OCR IMPLEMENTATION
=====================

Primary OCR module
------------------
`modules/ocr_utils.py` is now the real OCR entry point.

Key behavior:
- EasyOCR reader is cached once in `_get_reader()`
- Language: English only (`['en']`)
- GPU disabled (`gpu=False`)
- Default confidence threshold used by the endpoint: `0.3`

Two-pass OCR strategy
---------------------
`extract_ingredients_from_image(image_bytes, confidence_threshold=0.3)` works in 2 passes:

Pass 1:
- Run OCR on the original image first

Pass 2:
- If no ingredient header is found, preprocess and run OCR again

Preprocessing steps:
- decode image bytes to OpenCV BGR
- grayscale
- CLAHE contrast enhancement
- mild denoising
- 2x upscale with cubic interpolation

Return behavior:
- If pass 1 finds an ingredient header, return pass 1 text
- Else if pass 2 finds an ingredient header, return pass 2 text
- Else return both OCR outputs combined for diagnostics


3. INGREDIENT SECTION PARSING
=============================

Parser location
---------------
- `modules/ocr_utils.extract_ingredient_section(raw_text)`

Parsing logic
-------------
The parser:
1. Looks for a fuzzy ingredient header
2. Accepts OCR-typo variants such as missing letters or `lngredients`
3. Falls back to scanning the full OCR text if no header is found
4. Cuts off parsing at stop-keywords such as:
   - `Marketed`
   - `Manufactured`
   - `Directions`
   - `Net Weight`
   - `Batch`
   - `MRP`
   - several address/manufacturer markers
5. Splits on commas and the word `and`
6. Cleans noise and short tokens

Output:
- Returns `list[str]` of ingredient names
- Returns `[]` if nothing usable is found


4. ANALYSIS LAYER AFTER OCR
===========================

Skin condition source
---------------------
`modules/history_db.py` maps the user's latest face-scan result to:
- `acne`
- `oily_skin`
- `dry_skin`
- `dark_spots`
- `normal_skin`

If no prior face scan exists, the product analysis falls back to:
- `general`

LLM path
--------
`modules/llm_provider.py` contains a live OpenRouter call:
- endpoint: `https://openrouter.ai/api/v1/chat/completions`
- model id: `google/gemini-flash-1.5`
- env var required: `OPENROUTER_API_KEY`

If the request succeeds, the backend tries to parse JSON with:
- `score`
- `good_ingredients`
- `bad_ingredients`
- `recommendation`

Rule-based fallback
-------------------
If the LLM is unavailable or parsing fails, the backend falls back to:
- `modules/product_scanner.analyze_custom_ingredients(...)`

That scoring is based on `data/ingredient_db.json`.


5. PER-INGREDIENT BREAKDOWN
===========================

`analysis_server.py` also builds `ingredient_breakdown`, where each ingredient gets:
- `name`
- `category`
- `rating` = `good`, `bad`, or `neutral`
- `reason`

The rating priority is:
1. condition-specific allow/avoid tables in `analysis_server.py`
2. LLM or rule-engine good/bad ingredient matches
3. static category metadata
4. neutral fallback


6. DEBUG AND FAILURE MODES
==========================

Debug mode
----------
`/api/analyze-product?debug=true` may include:
- `ocr_raw`
- `ocr_ingredients_raw`

Failure behavior
----------------
If no ingredient section is found:
- API returns HTTP 400
- A user-facing retry message is returned
- OCR raw text is appended to `SCANNER.log`

Success logging
---------------
`SCANNER.log` is also appended on pass/success states.


7. TECHNICAL TRUTH SUMMARY
==========================

What is true right now
----------------------
- OCR is active and wired through `modules/ocr_utils.py`
- OCR uses a two-pass strategy, not just one raw EasyOCR call
- Product analysis is condition-aware based on the user's latest skin scan
- OpenRouter/Gemini integration now exists in executable runtime code
- Rule-based scoring still exists as the fallback safety path

What is no longer true
----------------------
- `/api/analyze-product` does not do the old inline OCR-only flow anymore
- product analysis is no longer permanently hardcoded to `"acne"`
- OpenRouter is not merely "configured"; it is implemented, but optional and failure-tolerant

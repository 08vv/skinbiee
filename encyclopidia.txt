SKINBIEE PROJECT ENCYCLOPEDIA
=============================

1. OVERVIEW
-----------
Skinbiee is a skincare companion app that combines:
- skin-condition prediction from face images
- OCR-based ingredient extraction from product labels
- compatibility scoring based on the user's latest detected skin condition
- planner and streak tracking features
- a mascot-style helper UI

The current working product is centered around the custom frontend in `frontend/`
and the Flask backend in `analysis_server.py`.

3. ACTIVE APPLICATION PATH
--------------------------
Frontend:
- `frontend/skinbiee.html`
- `frontend/skinbiee.js`
- `frontend/skinbiee.css`

Frontend server:
- `serve_skinbiee.py`
- serves on port `8001`

Backend:
- `analysis_server.py`
- serves on port `5000`

The frontend calls the backend for:
- face analysis
- product analysis
- daily logs
- user history / streak data


4. BACKEND CAPABILITIES
-----------------------
Authentication:
- username/password registration
- username/password login

Face analysis:
- classifies image into:
  - Acne
  - Dark Spots
  - Oiliness
  - Dryness
  - Normal

Product analysis:
- uploads product photo
- runs OCR
- extracts ingredient list
- finds the user's latest face-scan condition
- analyzes ingredients with LLM or rules
- returns structured ingredient results

Planner support:
- stores AM/PM completion, notes, ratings, and optional photo

User dashboard support:
- returns scans, logs, streak, and active calendar dates


5. OCR AND INGREDIENT ENGINE
----------------------------
OCR is implemented in `modules/ocr_utils.py`.

Current behavior:
- EasyOCR reader is cached
- OCR runs on the original image first
- if no ingredient header is found, OCR runs again on a preprocessed image
- ingredient parsing uses fuzzy header matching and stop-keyword trimming

This is more advanced than the earlier single-pass OCR flow.

Ingredient scoring sources:
- primary attempt: `modules/llm_provider.py`
- fallback: `modules/product_scanner.py`
- rule data: `data/ingredient_db.json`


6. OPENROUTER STATUS
--------------------
OpenRouter is now an implemented runtime dependency for ingredient analysis when:
- `OPENROUTER_API_KEY` is present
- the external request succeeds

Model currently referenced:
- `google/gemini-flash-1.5`

If the call fails, the backend falls back to deterministic rule-based scoring.

So the correct status is:
- not mandatory
- not the main chatbot engine
- but actively wired into the product-analysis path


7. CHATBOT STATUS
-----------------
The mascot chat experience is still mostly rule-based.

Frontend chat:
- `frontend/skinbiee.js`
- keyword matching and templated replies

Backend chatbot module:
- `modules/mascot_chatbot.py`
- intent/keyword responses from `data/chat_data.json`

This means the app has both:
- rule-based chat
- optional LLM ingredient analysis

Those are separate systems.


8. DATA LAYERS
--------------
Database tables:
- `users`
- `scans`
- `daily_logs`
- `routines`

Storage behavior:
- PostgreSQL if configured by `DATABASE_URL`
- SQLite fallback otherwise

Media storage:
- Cloudinary for uploaded scan and progress images

Knowledge/data files:
- `data/chat_data.json`
- `data/ingredient_db.json`

9. IMPORTANT ACCURACY NOTES
---------------------------
- The custom frontend is the clearest active product path in this repo
- The Streamlit app entrypoint exists, but supporting `ui` / `pages` files are missing here
- Some old documentation and chatbot copy still describe a more local-only or purely rule-based system
- The latest backend now includes Cloudinary uploads and optional OpenRouter usage


10. SHORT ENCYCLOPEDIA SUMMARY
------------------------------
Skinbiee is currently a hybrid skincare app:
- TensorFlow for face classification
- EasyOCR for label reading
- regex/rule logic for ingredient extraction and fallbacks
- optional OpenRouter/Gemini for richer ingredient analysis
- Flask API plus Vanilla JS frontend for the main user experience

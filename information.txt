SKINBIEE - CURRENT PROJECT DOCUMENTATION
========================================

Last updated
------------
This document reflects the code currently present in the repository on 2026-04-05.


1. PROJECT SUMMARY
==================

Skinbiee is a skincare tracking and analysis app with three main capabilities:
- face-scan based skin-condition prediction
- product ingredient scanning through OCR
- routine logging, streak tracking, and mascot-guided help

The active product experience is the custom web frontend in `frontend/`.
The Python backend lives in `analysis_server.py`.


2. ACTIVE RUNTIME ARCHITECTURE
==============================

Frontend
--------
- Main UI: `frontend/skinbiee.html`
- Main logic: `frontend/skinbiee.js`
- Styling: `frontend/skinbiee.css`
- Served locally by: `serve_skinbiee.py`
- Default frontend URL: `http://127.0.0.1:8001/skinbiee.html`

Backend API
-----------
- Main API server: `analysis_server.py`
- Framework: Flask + CORS
- Default API URL: `http://127.0.0.1:5000`
- Production serving preference: Waitress if installed, otherwise Flask dev server

Media and persistence
---------------------
- Uploaded images are stored in Cloudinary when credentials are configured
- User data is stored in:
  - PostgreSQL if `DATABASE_URL` points to Postgres
  - otherwise local SQLite at `data/skin_history.db`

4. CURRENT API ENDPOINTS
========================

`GET /`
-------
- health check
- returns backend status and DB mode

`POST /api/auth/register`
-------------------------
- body: JSON
- fields: `username`, `password`
- returns created user id and username on success

`POST /api/auth/login`
----------------------
- body: JSON
- fields: `username`, `password`
- returns user id and username on success

`POST /api/analyze-skin`
------------------------
- multipart form with `image` and `user_id`
- uploads image to Cloudinary
- preprocesses to `224x224`
- runs TensorFlow model prediction
- returns condition list with confidence values
- stores top result in `scans`

Runtime labels:
- `Acne`
- `Dark Spots`
- `Oiliness`
- `Dryness`
- `Normal`

`POST /api/analyze-product`
---------------------------
- multipart form with `image` and `user_id`
- uploads image to Cloudinary
- runs OCR through `modules/ocr_utils.py`
- extracts ingredient list
- fetches user's latest skin condition from DB
- tries OpenRouter/Gemini analysis first
- falls back to rule-based ingredient scoring if needed
- returns:
  - `skin_condition`
  - `ingredients_detected`
  - `ingredient_breakdown`
  - `analysis`
  - `image_url`

Debug support:
- `?debug=true` may include `ocr_raw` and `ocr_ingredients_raw`

`POST /api/daily-log`
---------------------
- multipart form with routine completion and optional image
- upserts a daily log row
- optional selfie uploads to Cloudinary

`GET /api/user/data`
--------------------
- query param: `user_id`
- returns:
  - `scans`
  - `logs`
  - `streak`
  - `active_dates`


5. FACE ANALYSIS PIPELINE
=========================

Implementation
--------------
- model path: `models/skin_model.h5`
- if loading fails, `analysis_server.py` uses a `MockModel`
- image preprocessing uses Pillow + OpenCV + NumPy

Output behavior
---------------
- predictions over `0.4` confidence are returned
- if nothing passes threshold, a fallback `Normal` result is returned
- scan record is persisted with timestamp and uploaded image URL

Important note
--------------
The bundled model appears to be a mock/demo model rather than a clinically validated one.
This project should be treated as a guidance/demo tool, not a medical system.


6. PRODUCT OCR AND INGREDIENT ANALYSIS
======================================

OCR
---
- powered by EasyOCR
- implemented in `modules/ocr_utils.py`
- uses a shared cached reader
- runs a two-pass strategy:
  - original image
  - preprocessed image if needed

Ingredient extraction
---------------------
- fuzzy header matching for `Ingredients`
- stop-keyword based trimming
- tokenization into clean ingredient names

Analysis behavior
-----------------
- condition comes from the user's latest face scan via `get_user_skin_condition(...)`
- LLM path uses OpenRouter with model `google/gemini-flash-1.5`
- fallback path uses `modules/product_scanner.py`
- result includes top-level score/recommendation and per-ingredient breakdown


7. CHATBOT / NLP STATUS
=======================

Frontend chatbot
----------------
`frontend/skinbiee.js` contains `getMascotAIResponse(...)`, which is rule-based.
It uses keyword matching and templated responses.

Backend chatbot module
----------------------
`modules/mascot_chatbot.py` is also rule-based and reads `data/chat_data.json`.

Important distinction
---------------------
- General mascot chat is still deterministic and keyword-driven
- LLM use currently appears in the ingredient-analysis path, not as the main chat engine


8. DATABASE BEHAVIOR
====================

Tables created by `modules/history_db.py`
-----------------------------------------
- `users`
- `scans`
- `daily_logs`
- `routines`

Notable behavior
----------------
- automatic Postgres-to-SQLite fallback
- `add_daily_log(...)` performs update-or-insert behavior by user/date
- `get_user_skin_condition(...)` ignores rows where `condition = 'Product Scan'`
- face-scan labels are normalized to ingredient-db keys

10. PRACTICAL TRUTH
===================

If you want the shortest accurate description of the current system:

- the real app is the `frontend/` SPA talking to `analysis_server.py`
- skin scans use TensorFlow classification
- product scans use EasyOCR plus ingredient parsing
- ingredient analysis is LLM-first when configured, rule-based otherwise
- user history and streaks come from the database
- some legacy or incomplete Streamlit pieces still exist in the repo, but they are not the main active path
